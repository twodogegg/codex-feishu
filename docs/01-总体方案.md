# codex-feishu 总体方案

## 1. 产品定位

`codex-feishu` 是一个面向团队的飞书控制面，用来把 Codex 的线程式工作流接入飞书消息与卡片交互。

第一版目标不是做一个“飞书版终端模拟器”，而是做一个：

- 只支持飞书入口
- 支持单机长期运行
- 每个用户拥有自己的持久 workspace
- 通过 `codex app-server` 承载实际执行
- 通过飞书卡片和命令完成日常操作

## 2. 核心设计原则

### 2.1 Workspace First

系统的核心资源是 `workspace`，不是 `repo`。

原因：

- 用户真正使用的是自己的工作目录
- `codex app-server` 天然围绕 `thread + cwd` 工作
- Git 仓库只是 workspace 的一种来源，不应该成为系统主语
- 后续可以支持空白 workspace、模板 workspace、导入型 workspace

### 2.2 单机服务，用户私有执行环境

部署形态为一套单机长期运行服务，但每个用户拿到的是自己的持久 workspace。

这意味着：

- 团队共用一个飞书机器人和这台机器上的单实例服务
- 每个用户有自己独立的 workspace 目录
- 每个 workspace 下维护自己的 Codex thread 列表
- 用户之间默认不共享未提交修改

### 2.3 App Server 作为执行内核

`codex app-server` 负责：

- 创建线程
- 恢复线程
- 列出线程
- 启动 turn
- 中断 turn
- 提供模型、skills、review、compact 等执行能力

平台自身负责：

- 用户与权限
- workspace 生命周期
- 飞书会话绑定
- 卡片渲染
- 审批与审计
- worker 管理

## 3. 可行性结论

基于本机 `codex-cli 0.120.0` 的 `app-server` 协议核实，当前版本已经具备以下关键能力：

- `thread/start` 支持 `cwd`
- `turn/start` 支持 `cwd`
- `thread/resume`
- `thread/list` 支持按 `cwd` 过滤
- `thread/read`
- `turn/interrupt`
- `skills/list`
- `review/start`
- `thread/compact/start`

因此，下面这些产品能力是可做的：

- 每个用户一个持久 workspace
- 每个 workspace 多个 thread
- 在飞书里切 workspace、切 thread、继续当前 thread
- 通过飞书执行 review、compact、skills、status 等兼容命令

## 4. 系统边界

### 4.1 第一版包含

- 飞书消息接入
- 飞书卡片交互
- 单机长期运行
- 个人持久 workspace
- workspace 与 thread 的绑定与恢复
- 命令兼容层
- 基础审批
- 基础审计

### 4.2 第一版不包含

- Telegram / Slack / Web 控制台
- 多人共享同一个 workspace
- 仓库托管平台集成作为主模型
- 多节点与分布式调度
- 完整终端 UI 语义复刻

## 5. 系统架构

整体采用“单机单实例控制面 + 本机 worker 管理”的模块化单体方案。

### 5.1 控制面

控制面负责：

- 接收飞书事件
- 管理用户
- 管理 workspace
- 管理会话绑定
- 管理命令路由
- 管理卡片回写
- 管理审批和审计

### 5.2 执行面

执行面通过 `Codex Worker Manager` 管理多个 workspace worker。

设计原则：

- 每个活跃 workspace 对应一个 worker
- 一个 worker 内部运行一个 `codex app-server`
- 该 worker 只负责自己的 workspace

在单机场景下这样做的好处：

- 隔离用户运行状态
- 降低线程和审批串扰风险
- 便于做超时、重启、回收
- 便于后续演进到更复杂部署

## 6. 主要模块

### 6.1 Feishu Gateway

负责：

- 订阅 `im.message.receive_v1`
- 订阅 `card.action.trigger`
- 发送消息
- 回复消息
- 更新卡片
- 上传文件
- 表情回复

### 6.2 Workspace Service

负责：

- 创建 workspace
- 自动创建用户默认 workspace
- 管理员预创建 workspace
- 启用 / 禁用 / 冻结 workspace
- 查询用户可用 workspace 列表

### 6.3 Session Router

负责：

- 维护“飞书会话当前绑定到哪个 workspace”
- 维护“当前 workspace 正在使用哪个 thread”
- 处理切换逻辑

### 6.4 Codex Worker Manager

负责：

- 拉起 worker
- 查找 worker
- 回收空闲 worker
- 重启异常 worker

### 6.5 Codex Worker

负责：

- 调用 `codex app-server`
- 管理当前 workspace 的线程与 turn
- 将流式事件回传控制面

### 6.6 Card System

负责：

- 状态卡
- 回复卡
- 审批卡
- workspace 卡

卡片统一使用 Feishu Card JSON 2.0。

## 7. 第一版典型链路

### 7.1 普通消息

1. 用户在飞书发送消息
2. Gateway 收到事件
3. Session Router 找到当前 workspace
4. Worker Manager 找到该 workspace 对应的 worker
5. Worker 恢复或创建 thread
6. Worker 发起 `turn/start`
7. 控制面将流式结果渲染为回复卡

### 7.2 卡片动作

1. 用户点击飞书卡片
2. Gateway 提取 action payload
3. Orchestrator 将动作路由到 workspace / thread / codex 能力
4. 更新状态卡或回复卡

### 7.3 审批

1. Worker 执行过程中产生审批请求
2. 控制面生成审批卡
3. 用户在飞书中批准或拒绝
4. 控制面把结果回传给 worker

## 8. 技术选型建议

- 语言：Node.js
- 运行时：单机长期在线服务
- 数据库：SQLite
- 缓存：第一版使用进程内内存缓存
- 卡片：Feishu Card JSON 2.0
- 执行器：`codex app-server`

## 9. 外部依赖

  https://open.feishu.cn/document/server-docs/im-v1/introduction

- 飞书卡片 JSON 2.0  
  https://open.feishu.cn/document/feishu-cards/card-json-v2-structure

## 10. 一句话结论

`codex-feishu` 的正确实现方式不是“把 Codex 终端搬进飞书”，而是“以 workspace 为核心的飞书控制面 + 以 `codex app-server` 为核心的本机 per-workspace 执行器”。
