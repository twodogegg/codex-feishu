# 数据库表设计

## 1. 目标

本文档把前面的数据模型进一步落成数据库表结构建议，方便后续直接建表和写 ORM 模型。

第一版默认使用 SQLite。

## 2. 设计原则

- 长期状态入库
- 高频运行态优先放进程内内存
- 表结构优先服务 MVP，不追求一次性做全

## 3. 表清单

第一版建议建 8 张核心表：

- `users`
- `workspace_templates`
- `workspaces`
- `threads`
- `session_bindings`
- `runs`
- `approval_rules`
- `card_messages`

## 4. 表结构建议

## 4.1 users

用途：

- 保存飞书用户映射

建议字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | text | 主键，建议使用 UUID 字符串 |
| feishu_open_id | text unique | 飞书 open_id |
| feishu_union_id | text | 飞书 union_id |
| feishu_user_id | text | 飞书 user_id |
| display_name | text | 显示名 |
| status | text | active / disabled |
| created_at | text | ISO 时间字符串 |
| updated_at | text | ISO 时间字符串 |

索引建议：

- `unique(feishu_open_id)`

## 4.2 workspace_templates

用途：

- 自动创建 workspace 时提供模板

建议字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | text | 主键 |
| name | text | 模板名 |
| description | text | 描述 |
| default_model | text | 默认模型 |
| default_effort | text | 默认 effort |
| policy_json | text | 默认策略 JSON 字符串 |
| created_at | text | ISO 时间字符串 |
| updated_at | text | ISO 时间字符串 |

## 4.3 workspaces

用途：

- 保存用户持久 workspace

建议字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | text | 主键 |
| owner_user_id | text | 所属用户 |
| template_id | text null | 来源模板 |
| name | text | workspace 名称 |
| slug | text | 稳定别名 |
| root_path | text | 服务器目录路径 |
| status | text | created / ready / active / idle / disabled / error |
| default_model | text | 默认模型 |
| default_effort | text | 默认 effort |
| policy_json | text | workspace 级策略 JSON 字符串 |
| last_active_thread_id | text null | 最近激活 thread |
| created_at | text | ISO 时间字符串 |
| updated_at | text | ISO 时间字符串 |

索引建议：

- `index(owner_user_id)`
- `unique(owner_user_id, slug)`

## 4.4 threads

用途：

- 保存平台侧 thread 元数据

建议字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | text | 主键 |
| workspace_id | text | 所属 workspace |
| codex_thread_id | text unique | app-server threadId |
| name | text null | thread 标题 |
| kind | text | main / subagent / review / compact |
| parent_thread_id | text null | 父 thread |
| status | text | created / running / waiting_approval / completed / failed / archived |
| is_active | boolean | 是否当前激活 |
| last_turn_id | text null | 最近 turnId |
| metadata_json | text | 额外元数据 JSON 字符串 |
| created_at | text | ISO 时间字符串 |
| updated_at | text | ISO 时间字符串 |

索引建议：

- `index(workspace_id)`
- `index(parent_thread_id)`

## 4.5 session_bindings

用途：

- 保存飞书会话当前绑定关系

建议字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | text | 主键 |
| user_id | text | 用户 |
| chat_id | text | 飞书 chat_id |
| thread_key | text null | 飞书话题 key |
| workspace_id | text null | 当前绑定 workspace |
| active_thread_id | text null | 当前激活 thread |
| created_at | text | ISO 时间字符串 |
| updated_at | text | ISO 时间字符串 |

索引建议：

- `index(user_id)`
- `index(chat_id)`
- `unique(user_id, chat_id, coalesce(thread_key, ''))`

## 4.6 runs

用途：

- 保存一次 turn 执行记录

建议字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | text | 主键 |
| workspace_id | text | 所属 workspace |
| thread_id | text | 所属 thread |
| turn_id | text null | app-server turnId |
| state | text | queued / running / waiting_approval / completed / failed / cancelled |
| reply_message_id | text null | 飞书回复消息 ID |
| started_at | text null | ISO 时间字符串 |
| first_token_at | text null | ISO 时间字符串 |

| metrics_json | text | 时延等指标 JSON 字符串 |

索引建议：

- `index(workspace_id)`
- `index(thread_id)`
- `index(state)`

## 4.7 approval_rules

用途：

- 保存 workspace 级自动审批规则

建议字段：

| 字段 | 类型 | 说明 |

| scope | text | once / workspace |

| updated_at | text | ISO 时间字符串 |

索引建议：

## 4.8 card_messages

用途：

- 保存飞书卡片与系统对象映射

建议字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| id | text | 主键 |
| chat_id | text | 飞书 chat_id |
| message_id | text unique | 飞书 message_id |
| workspace_id | text null | 关联 workspace |
| thread_id | text null | 关联 thread |

| created_at | text | ISO 时间字符串 |
| updated_at | text | ISO 时间字符串 |

## 5. 进程内运行态缓存建议

不建议把所有运行态都放数据库。

第一版建议放进程内内存的内容：

- worker 状态
- 当前活跃 run
- `threadId -> turnId`
- reply card flush 定时状态
- 当前审批请求

如果后续改成多实例，再迁移到 Redis。

## 6. 约束建议

### 6.1 workspace 约束

- 一个用户可以有多个 workspace
- 同一个用户下 `slug` 必须唯一

### 6.2 thread 约束

- `codex_thread_id` 全局唯一
- sub-agent thread 必须有 `parent_thread_id`

### 6.3 session 约束

- 同一用户同一飞书会话同一话题，只应有一条绑定记录

## 7. 删除策略

第一版建议：

- 用户表：逻辑删除或禁用
- workspace：逻辑删除或禁用
- thread：不物理删除，允许保留历史
- runs：长期保留

## 8. 迁移顺序建议

建议按顺序建表：

1. `users`
2. `workspace_templates`
3. `workspaces`
4. `threads`
5. `session_bindings`
6. `runs`
7. `approval_rules`
8. `card_messages`

## 9. 开发建议

第一版写代码时，建议：

- 先有显式 repository 层
- 不要把 SQL 散落在飞书事件处理器里
- `thread` 与 `run` 更新要放事务

## 10. 一句话结论

数据库的职责是保存“可恢复的长期状态”，而不是替代 `app-server` 保存 thread 内部历史。第一版数据库选型为 SQLite，本机单实例足够。
