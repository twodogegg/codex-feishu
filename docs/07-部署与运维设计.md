# 部署与运维设计

## 1. 目标

本文档说明第一版如何在单机环境中部署、运行、观测和恢复 `codex-feishu`。

重点关注：

- 单机长期在线服务部署
- workspace worker 管理
- 故障恢复
- 资源控制

## 2. 部署形态

第一版建议部署为单机单实例长期在线服务。

组成：

- `codex-feishu api`
- `sqlite database`
- `workspace storage`
- `codex worker processes`

## 3. 推荐运行环境

建议：

- macOS 或 Linux
- Node.js 20+
- SQLite 3
- 安装 `codex-cli 0.120.0+`

## 4. 目录规划

建议目录：

```text
/srv/codex-feishu/
  app/
  config/
  logs/
  tmp/
  data/
    app.db
  workspaces/
    user_001/
      ws_default/
    user_002/
      ws_default/
```

原则：

- workspace 目录独立
- 日志与工作区分离
- 程序目录与数据目录分离

## 5. 服务进程

### 5.1 主服务

负责：

- 飞书事件接入
- 命令路由
- 卡片更新
- 数据库读写
- worker 调度

### 5.2 Worker 进程

负责：

- 承载单个 workspace 的 `codex app-server`
- 管理该 workspace 的 thread / turn

建议策略：

- 一个活跃 workspace 一个 worker
- worker 无活跃使用时允许回收
- 全部 worker 由主进程管理

## 6. Worker 生命周期

建议状态：

- `starting`
- `ready`
- `busy`
- `idle`
- `stopping`
- `error`

### 6.1 启动

触发条件：

- 用户进入某 workspace
- 该 workspace 当前没有 worker

动作：

- 启动 `codex app-server`
- 执行 `initialize`
- 标记 worker 为 `ready`

### 6.2 运行

普通消息到达时：

- `ready -> busy`
- turn 完成后
- `busy -> idle`

### 6.3 回收

建议：

- 空闲超过指定时间自动回收
- 默认 10 到 30 分钟

### 6.4 恢复

worker 被回收后：

- workspace 数据不丢
- 下次进入时重新拉起
- thread 通过 `thread/resume` 恢复

## 7. 资源控制

第一版至少控制：

- 每用户最大 workspace 数
- 每 workspace 同时活跃 turn 数
- 单机最大 worker 数
- worker 空闲超时时间
- 卡片更新频率

建议默认：

- 每 workspace 同时仅 1 个活跃 turn
- worker 空闲 15 分钟回收
- SQLite 只承载长期状态写入

## 8. 日志设计

建议日志分层：

- 访问日志
- 命令日志
- worker 日志
- 审批日志
- 卡片更新日志

建议字段：

- `request_id`
- `user_id`
- `workspace_id`
- `thread_id`
- `run_id`
- `chat_id`
- `action`
- `status`
- `duration_ms`

## 9. 审计设计

必须记录：

- 用户执行了什么命令
- 操作了哪个 workspace
- 使用了哪个 thread
- 发起了哪个 review / compact / stop
- 哪次审批被允许或拒绝

第一版可以先落数据库，不要求独立审计系统。

## 10. 监控指标

建议监控：

- 活跃用户数
- 活跃 workspace 数
- 活跃 worker 数
- turn 成功率
- turn 平均耗时
- 卡片 patch 次数
- worker 启动失败次数
- 审批请求次数

## 11. 故障恢复策略

### 11.1 主服务重启

要求：

- 不丢数据库状态
- 不丢 workspace 元数据
- worker 直接重新拉起，不要求原进程延续

### 11.2 Worker 崩溃

要求：

- 当前 run 标记失败
- 当前用户收到失败提示
- 下次消息自动重启 worker

### 11.3 飞书卡片更新失败

要求：

- 记录日志
- 可重试有限次数
- 超出次数后退化为提示消息

## 12. 配置建议

建议环境变量分类：

- 飞书配置
- SQLite 路径配置
- worker 配置
- workspace 路径配置
- 默认模型与 effort 配置

## 13. 上线前检查清单

- 飞书事件订阅完成
- 飞书权限 scopes 配齐
- 卡片交互回调可用
- SQLite 数据库可创建
- `codex app-server` 可正常初始化
- worker 可拉起并回收

## 14. 第一版不做的运维能力

- 多节点调度
- 区域级容灾
- 自动扩缩容平台
- 完整审计后台

## 15. 一句话结论

第一版运维策略的核心是：

`主服务单机常驻，worker 按 workspace 拉起和回收，SQLite 保存所有可恢复的长期状态。`
