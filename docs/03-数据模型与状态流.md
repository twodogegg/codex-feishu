# 数据模型与状态流

## 1. 核心对象

第一版最重要的 5 个对象：

- `User`
- `Workspace`
- `Thread`
- `SessionBinding`
- `Run`

## 2. 数据模型

### 2.1 User

表示飞书用户。

建议字段：

- `id`
- `feishu_open_id`
- `feishu_union_id`
- `feishu_user_id`
- `display_name`
- `status`
- `created_at`
- `updated_at`

### 2.2 Workspace

表示用户私有持久工作区。

建议字段：

- `id`
- `owner_user_id`
- `name`
- `slug`
- `root_path`
- `status`
- `template_id`
- `default_model`
- `default_effort`
- `policy_id`
- `last_active_thread_id`
- `created_at`
- `updated_at`

说明：

- `Workspace` 是一等资源
- 一个用户可以有多个 workspace
- 一个 workspace 长期存在，不因飞书会话结束而销毁

### 2.3 Thread

表示某个 workspace 下的一条 Codex thread。

建议字段：

- `id`
- `workspace_id`
- `codex_thread_id`
- `name`
- `kind`
- `parent_thread_id`
- `is_active`
- `status`
- `last_turn_id`
- `created_at`
- `updated_at`

说明：

- `kind` 可取：
  - `main`
  - `subagent`
  - `review`
  - `compact`

### 2.4 SessionBinding

表示飞书会话与系统状态的绑定。

建议字段：

- `id`
- `user_id`
- `chat_id`
- `thread_key`
- `workspace_id`
- `active_thread_id`
- `created_at`
- `updated_at`

说明：

- 一个飞书会话当前只绑定一个 active workspace
- 一个 active workspace 当前只绑定一个 active thread

### 2.5 Run

表示一次运行记录。

建议字段：

- `id`
- `workspace_id`
- `thread_id`
- `turn_id`
- `state`
- `reply_message_id`
- `started_at`
- `first_token_at`
- `ended_at`
- `error_text`

## 4. 关系说明

### 4.1 用户与 workspace

- 一个 `User` 可以有多个 `Workspace`
- 一个 `Workspace` 只属于一个 `User`

### 4.2 workspace 与 thread

- 一个 `Workspace` 可以有多个 `Thread`
- 一个 `Thread` 只属于一个 `Workspace`

### 4.3 thread 与 sub-agent

- sub-agent thread 仍然是 `Thread`
- 只是 `kind=subagent`
- 通过 `parent_thread_id` 指向主 thread

这样设计的好处：

- 不用额外发明新的 agent 表
- `/subagents` 可以复用 thread 列表体系
- review / compact 也可以复用同一套对象

## 5. Workspace 生命周期

建议状态：

- `created`
- `ready`
- `active`
- `idle`
- `disabled`
- `error`

状态含义：

- `created`
  - 记录已创建，但目录还未准备完成
- `ready`
  - 可进入但当前没有活跃 worker
- `active`
  - 有活跃 worker 或当前正在使用
- `idle`
  - 暂时空闲，可被回收 worker
- `disabled`
  - 已禁用，不允许进入
- `error`
  - 工作区损坏或初始化失败

## 6. Thread 生命周期

建议状态：

- `created`
- `running`
- `waiting_approval`
- `completed`
- `failed`
- `archived`

## 7. 关键状态流

### 7.1 首次使用自动创建

1. 用户首次给机器人发消息
2. 系统查找该用户默认 workspace
3. 若不存在，则根据模板自动创建
4. workspace 进入 `ready`
5. 会话绑定建立
6. 若用户继续发送普通消息，则创建第一条 thread

### 7.2 进入已有 workspace

1. 用户执行 `/bind <workspace>`
2. 更新 `SessionBinding.workspace_id`
3. 读取该 workspace 的 `last_active_thread_id`
4. 同步状态卡

### 7.3 普通消息执行

1. 找到 active workspace
2. 找到 active thread
3. 没有则创建新 thread
4. 创建 `Run`
5. 执行 `turn/start`
6. 流式更新卡片
7. `Run` 最终进入 `completed` 或 `failed`

### 7.4 切 thread

1. 用户执行 `/switch <threadId>`
2. 校验该 thread 是否属于当前 workspace
3. 更新 `SessionBinding.active_thread_id`
4. 更新 `Workspace.last_active_thread_id`

### 7.5 Sub-agent 切换

1. 用户执行 `/subagents`
2. 系统列出 `parent_thread_id = 当前主线程` 的子线程
3. 用户选择某个 sub-agent thread
4. 更新当前 session 的 `active_thread_id`

## 8. 与 App Server 的映射

### 8.1 Workspace 到 app-server

`app-server` 不管理 workspace，对应关系由平台自己保存。

映射原则：

- `workspace.root_path` -> thread / turn 的 `cwd`
- `thread.codex_thread_id` -> app-server 的 `threadId`

### 8.2 平台保存什么

平台保存：

- 用户和 workspace 的关系
- 哪个 thread 属于哪个 workspace
- 哪个飞书会话当前激活哪个 workspace / thread

### 8.3 App Server 保存什么

App Server 负责：

- thread 本体
- turn 本体
- thread 内部历史
- 模型运行时状态

## 9. 最小建表建议

第一版至少建：

- `users`
- `workspace_templates`
- `workspaces`
- `threads`
- `session_bindings`
- `runs`
- `approval_rules`
- `card_messages`

## 10. 一句话结论

第一版最重要的建模原则是：

`workspace 是核心资源，thread 是 workspace 的执行历史，sub-agent 只是特殊 kind 的 thread。`
