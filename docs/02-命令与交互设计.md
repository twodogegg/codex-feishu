# 命令与交互设计

## 1. 设计目标

命令设计遵循两个原则：

- 尽量保留现有 `codex-im` 用户习惯
- 在团队共享服务架构下，把语义改造成 `workspace-first`

用户看到的仍然像“绑定项目、切线程、查看状态”，但系统内部绑定的是用户自己的持久 workspace。

## 2. 命令分层

系统命令分为三类：

- `workspace 平台命令`
- `thread 命令`
- `codex 兼容命令`

## 3. 命令清单

### 3.1 Workspace 平台命令

#### `/bind <workspace>`

含义：

- 将当前飞书会话绑定到指定 workspace

注意：

- 第一版不再支持“用户本机绝对路径”
- 参数应是 workspace 名称、别名或 ID

#### `/where`

含义：

- 查看当前会话绑定的 workspace
- 查看当前 active thread
- 查看当前模型、effort、运行状态

#### `/workspace`

含义：

- 列出当前用户可见的 workspace
- 展示每个 workspace 的最近 thread 状态

#### `/remove <workspace>`

含义：

- 将指定 workspace 从当前会话解绑

注意：

- 只移除绑定，不删除真实 workspace

#### `/send <相对路径>`

含义：

- 发送当前 workspace 内的文件到飞书会话

限制：

- 只允许相对路径
- 必须落在当前 workspace 目录内

### 3.2 Thread 命令

#### `/message`

含义：

- 查看当前 thread 最近消息

#### `/switch <threadId>`

含义：

- 切换当前 workspace 下的 active thread

#### `/new`

含义：

- 在当前 workspace 下创建新 thread，并切换过去

#### `/rename <name>`

含义：

- 重命名当前 thread

#### `/stop`

含义：

- 中断当前运行中的 turn

#### `/subagents`

含义：

- 查看当前主 thread 下的 sub-agent threads
- 切换 active agent thread

飞书语义：

- 不做终端 tab 语义
- 改做“列出 agent threads 并切换”

建议扩展子命令：

- `/subagents`
- `/subagents switch <threadId>`
- `/subagents back`

### 3.3 Codex 兼容命令

#### `/model`

含义：

- 查看当前 workspace 默认模型

#### `/model update`

含义：

- 刷新可用模型列表

#### `/model <modelId>`

含义：

- 设置当前 workspace 默认模型

#### `/effort`

含义：

- 查看当前 workspace 默认推理强度

#### `/effort <low|medium|high|xhigh>`

含义：

- 设置当前 workspace 默认推理强度

#### `/status`

含义：

- 展示当前 session 综合状态

建议展示：

- 当前 workspace
- 当前 thread
- 模型
- effort
- 当前 worker 状态
- 账号状态
- token / rate limit 概况
- skills 数量

#### `/skills`

含义：

- 列出当前 workspace 可见 skills

#### `/review`

含义：

- 对当前 workspace 执行 code review

#### `/compact`

含义：

- 对当前 thread 发起 compact

#### `/permissions`

含义：

- 展示当前 workspace 执行权限策略
- 可选地支持切换有限的策略档位

#### `/experimental`

含义：

- 展示当前平台允许启用的实验特性

#### `/fast`

含义：

- 切换一个“快模式”档位
- 本质上是模型 / 服务档位快捷开关

#### `/statusline`

含义：

- 第一版不映射终端状态栏
- 若保留该命令，仅展示“状态卡显示项配置”

## 4. 命令到 App Server RPC 映射

### 4.1 直接映射

这些命令背后有明确可调用的 `app-server` 能力：

- `/message` -> `thread/read`
- `/switch` -> `thread/list` + `thread/resume`
- `/new` -> `thread/start`
- `/stop` -> `turn/interrupt`
- `/model` -> `model/list` + thread / turn 覆盖
- `/skills` -> `skills/list`
- `/review` -> `review/start`
- `/compact` -> `thread/compact/start`
- `/status` -> `config/read` + `account/read` + `account/rateLimits/read`
- `/rename` -> `thread/name/set`

### 4.2 平台层封装

这些命令需要平台自己实现：

- `/bind`
- `/where`
- `/workspace`
- `/remove`
- `/send`
- `/permissions`
- `/experimental`
- `/statusline`
- `/subagents`

其中 `/subagents` 虽然底层与 agent thread 有关系，但第一版仍建议通过平台层重新组织展示与切换体验。

## 5. 卡片交互设计

除文本命令外，建议为高频操作提供卡片按钮或下拉选择：

- 切 workspace
- 切 thread
- 新建 thread
- 停止运行
- 切模型
- 切 effort
- 打开 sub-agents 列表
- 审批通过 / 拒绝

## 6. Action Payload 规范

卡片交互统一采用结构化 payload：

```json
{
  "version": "1",
  "kind": "thread",
  "action": "switch",
  "workspaceId": "ws_xxx",
  "threadId": "thread_xxx"
}
```

建议字段：

- `version`
- `kind`
- `action`
- `workspaceId`
- `threadId`
- `requestId`

必须加 `version`，方便未来做兼容升级。

## 7. 返回样式

建议交互统一遵循：

- 命令执行成功：返回信息卡或状态卡
- 长过程执行：返回流式回复卡
- 配置类操作：返回状态卡
- 危险动作：返回审批卡
