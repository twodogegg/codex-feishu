# 飞书卡片与消息设计

## 1. 目标

本文档定义：

- 哪些场景用文本
- 哪些场景用卡片
- 卡片如何组织
- 卡片如何更新
- 卡片动作 payload 如何设计

第一版统一以 Feishu Card JSON 2.0 为基础。

## 2. 设计原则

### 2.1 卡片是主交互载体

在本产品里：

- 简单提示可以用文本卡片
- 状态、选择、审批、流式回复都使用卡片

原因：

- 飞书命令返回纯文本可读性差
- 线程切换、workspace 切换、审批都天然适合交互卡片

### 2.2 文本消息只承担补充角色

文本消息适合：

- 极短错误提示
- 调试阶段兜底输出

生产形态仍建议统一为卡片。

### 2.3 卡片统一使用 JSON 2.0

统一要求：

- `schema: "2.0"`
- `update_multi: true`
- 状态类卡片使用共享更新
- 长过程回复卡可启用 `streaming_mode`

参考：

- https://open.feishu.cn/document/feishu-cards/card-json-v2-structure

## 3. 卡片类型

第一版定义 5 类卡片：

- `info_card`
- `status_card`
- `workspace_card`
- `reply_card`
- `approval_card`

## 4. 各类卡片定义

## 4.1 Info Card

用途：

- 成功提示
- 失败提示
- 简短反馈

典型场景：

- `/bind` 成功
- `/remove` 成功
- 参数错误
- 文件不存在

内容结构：

- 标题
- 一段 markdown 文本

## 4.2 Status Card

用途：

- 展示当前会话状态

展示内容建议：

- 当前 workspace
- 当前 thread
- 当前模型
- 当前 effort
- 当前 worker 状态
- 当前 run 状态

建议动作：

- 切 workspace
- 查看 messages
- 新建 thread
- 停止运行
- 刷新状态

## 4.3 Workspace Card

用途：

- 展示当前用户可用的 workspace 列表

展示内容建议：

- workspace 名称
- 当前活跃 thread
- 更新时间
- 状态

建议动作：

- 绑定该 workspace
- 查看状态
- 移除当前会话绑定

## 4.4 Reply Card

用途：

- 展示 Codex 的流式回复

展示内容建议：

- 标题：处理中 / 已完成 / 执行失败
- 摘要预览
- 模型信息
- 首字耗时 / 总耗时
- 正文 markdown

建议动作：

- 停止
- 新建 thread
- 重试
- 查看 thread 状态

## 4.5 Approval Card

用途：

- 展示审批请求

展示内容建议：

- 请求类型
- 原因
- 命令
- 风险提示

建议动作：

- 本次允许
- 当前 workspace 自动允许
- 拒绝

## 5. Reply Card 流式更新策略

## 5.1 发送策略

第一版建议：

1. 收到首段输出后创建首张 reply card
2. 后续按节流批量 patch
3. 完成后 patch 为最终状态

## 5.2 节流策略

建议：

- 300ms 批量刷新一次
- 如果是首条内容，立即发卡
- 如果 turn 完成或失败，立即 patch

## 5.3 长文本策略

为了避免卡片过长：

- 生成中阶段可以只展示尾部窗口
- 完成后展示完整正文或截断正文
- 超长内容建议折叠前文并提示

## 6. 卡片动作 Payload

卡片交互统一使用结构化 payload。

示例：

```json
{
  "version": "1",
  "kind": "workspace",
  "action": "switch",
  "workspaceId": "ws_001"
}
```

建议字段：

- `version`
- `kind`
- `action`
- `workspaceId`
- `threadId`
- `runId`
- `requestId`
- `selectedValue`

## 7. Kind 设计

第一版建议使用以下 `kind`：

- `workspace`
- `thread`
- `panel`
- `approval`
- `callback`

## 8. 常见动作定义

### 8.1 Workspace

- `switch`
- `status`
- `remove`

### 8.2 Thread

- `switch`
- `messages`
- `rename`

### 8.3 Panel

- `new_thread`
- `stop`
- `show_messages`
- `set_model`
- `set_effort`
- `open_subagents`

### 8.4 Approval

- `approve`
- `reject`
- `approve_workspace`

## 9. 飞书消息使用场景

结合飞书 IM 能力，建议这样使用：

- 发送消息：创建第一张卡片
- 回复消息：将命令回执和普通对话回复到原消息
- 更新卡片：统一 patch 当前卡片
- 表情回复：作为“处理中”轻提示
- 发送文件：通过 `/send`

参考：

- https://open.feishu.cn/document/server-docs/im-v1/introduction

## 10. 卡片布局建议

统一布局规则：

- 头部显示状态与标题
- 第一屏必须能看到当前 workspace 或当前 thread 关键信息
- 操作按钮不要超过 3 个一组
- 重要操作放在卡片底部

## 11. 卡片样式建议

状态色建议：

- 处理中：蓝 / turquoise
- 已完成：绿
- 失败：红
- 审批：橙 / 黄

标题建议：

- 状态卡：`当前状态`
- 回复卡：`Codex 回复`
- 审批卡：`审批请求`
- workspace 卡：`我的 Workspaces`

## 12. 错误处理

卡片错误场景分三类：

- 平台错误
- worker 错误
- app-server 执行错误

呈现策略：

- 用户可理解的短标题
- 一段简短原因说明
- 可恢复动作按钮

示例：

- `执行失败`
- `当前线程不可恢复，已建议创建新线程`

## 13. 兼容命令的卡片化输出

建议统一输出：

- `/status` -> 状态卡
- `/skills` -> skills 列表卡
- `/review` -> review 结果卡
- `/subagents` -> agent thread 列表卡

## 14. 一句话结论

第一版交互设计的核心是：

`命令负责触发，卡片负责展示和继续操作。`
